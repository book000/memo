## プロジェクト概要

- プロジェクト名: book000/memo
- 目的: Linux 系や各種プログラミング言語、フレームワークなどの技術メモ（備忘録）を管理し、Web サイトとして公開する
- サイト URL: https://memo.tomacheese.com
- ライセンス: CC BY-SA 4.0
- 主な機能:
  - Markdown 形式での技術メモの記述
  - MkDocs による静的サイトの生成
  - textlint / markdownlint による文書品質の維持
  - GitHub Pages での自動デプロイ

## 開発コマンド

### 依存関係のインストール

```bash

# Python パッケージのインストール (MkDocs)
```

## 文書作成ガイドライン

### Markdown ファイル編集の基本フロー

1. 適切なカテゴリ (`docs/programming/`, `docs/os/` など) にファイルを配置
2. ファイル名は小文字とハイフンを使用 (例: `nodejs-project.md`)
3. 前書きとして目的や概要を記述
4. 実際のコード例やスクリーンショットを含める
5. コミット後、`pnpm fix` → `pnpm lint` → 手動修正の順で品質を担保

### 自動修正後の確認事項

`pnpm fix` によって自動修正されたことにより、文章として不成立な状態・意味不明な文章になっていないかはよく確認すること。

特に、"。" に置き換わるルールがあるが、これによって違和感のある文章となっていないかは確認する。

### 日本語文書の品質ルール

1. 句読点は句点（。）と読点（、）を正しく使用する
2. 表記統一は同じ意味の言葉で統一した表記を使用する
3. 助詞の重複では同じ助詞の連続使用を避ける
4. 敬語では「ですます」調と「である」調の混在を避ける
5. 半角・全角では英数字は半角、日本語は全角を基本とする
6. スペースでは全角文字と半角文字の間に必要なスペースを入れる

### Markdown 記法の推奨パターン

- 見出しレベルは h1 から順序良く使用する（h1 は 1 つのファイルに 1 つ）
- コードブロックには言語を指定する
- リンクは明確でわかりやすい形式で記述する
- 画像には alt テキストをつける

## AI らしい表現の禁止

このプロジェクトでは、AI が生成した文章に共通して見られる不自然な表現パターンを避けること。
textlint の `preset-ai-writing` ルールによって自動検出できる項目もあるが、以下の項目は手動で確認・修正すること。

参考: [AI 生成文の「AI くささ」を取り除く技術](https://zenn.dev/m0370/articles/205c9340a418c3)

### 接続詞の連続使用禁止

「さらに」「加えて」「また」「なお」「一方で」などの接続詞を連続して使わないこと。
同じ接続詞を繰り返し使うと AI 生成の痕跡が残る。段落構成で意味のつながりを示すこと。

### 付け足し構文の禁止

データや事実を提示した後に、「〜を浮き彫りにしており」「〜を示唆しています」「〜に大きな示唆を与えています」のような意義づけを付け足さないこと。
事実の提示だけで十分であり、読者が解釈する余地を残すこと。

### 回りくどい表現の禁止

「〜として位置づけられています」「〜の役割を果たしています」「〜として機能します」のような複雑な表現を使わないこと。
シンプルに「〜です」「〜である」で表現できる場合は直接書くこと。

### 三点セットの強制禁止

何でも 3 つの要素に無理やりまとめないこと。内容に応じて 2 つや 4 つ以上になる場合はそのまま記載すること。

### 定型的な書き出し・結論の禁止

「本記事では〜について解説します」「〜について詳しく見ていきましょう」のような定型的な書き出しや、
「今後の展開が注目されます」「引き続き活用していきましょう」のような定型的な結論を使わないこと。

### 曖昧な出典表現の禁止

「〜と言われています」「〜とされています」「専門家によると」のような曖昧な出典表現を使わないこと。
出典を明示できる場合はリンクや参照を示すこと。出典がない場合は断定か自身の見解として書くこと。

### 過剰なヘッジングの禁止

「〜かもしれません」「〜の可能性があります」「〜と思われます」「〜ではないでしょうか」のような逃げ道を作る表現を多用しないこと。
確信がある場合は断定で書くこと。不確かな場合は「〜は未確認」「〜は要確認」のように明示すること。

### チャットボット口調の禁止

「承知しました」「かしこまりました」「ご質問ありがとうございます」「以上、ご参考になれば幸いです」のような
チャットボット対話的な口調を使わないこと。技術文書として客観的に記述すること。

### 追従的・お世辞的表現の禁止

「おっしゃる通りです」「素晴らしいご質問です」「おっしゃる通り〜ですね」のような過度に同調する表現を使わないこと。

### 具体性のある記述

「高速」「効率的」「簡単」「使いやすい」のような抽象的な形容を使う場合は具体的な数値や条件を示すこと。
個人的な経験や実際の動作確認に基づいた記述を心がけること。

### textlint による自動チェック

以下のパターンは textlint の `preset-ai-writing` ルールで自動検出される。

- インラインヘッダー付き箇条書き（`- **項目名:** 説明` 形式）
- 誇張表現・ハイプ表現
- 太字の機械的多用
- 冗長表現（「〜することができます」→「〜できます」）※ `preset-ja-technical-writing` の `ja-no-redundant-expression` ルールでも検出される

`pnpm lint` でチェックし、`pnpm fix` で自動修正すること。

## リポジトリ固有

### MkDocs について

- このプロジェクトは MkDocs with Material theme を使用して静的サイトを生成します
- `mkdocs.yml` でサイト構造、テーマ、プラグイン、拡張機能を設定しています
- GitHub Pages に自動デプロイされます

### textlint について

- textlint は日本語文書の品質を保つためのツールです
- `.textlintrc` でルールを設定しています
- 以下のルールが設定されています:
  - い抜き言葉の検出
  - フィラー（つなぎ表現）の禁止
  - サ抜き、サ入れ表現の誤用チェック
  - れ足す言葉のチェック
  - 同義語の表記ゆれチェック
  - TODO コメントの禁止
  - 日付と曜日の不一致の検出
  - ひらがな表記の推奨
  - 全角と半角アルファベットの混在チェック
  - スペース周りのスタイルチェック
  - 技術文書向けルール

### markdownlint について

- markdownlint は Markdown 記法の品質を保つためのツールです
- `.markdownlint.jsonc` でルールを設定しています
- 見出し、リスト、コードブロック、リンクなどの記法をチェックします

### Visual Studio Code 連携

- Visual Studio Code でこのプロジェクトを開くと、venv 環境下で mkdocs の serve が行われ、Chrome で `localhost:8080` が開きます
